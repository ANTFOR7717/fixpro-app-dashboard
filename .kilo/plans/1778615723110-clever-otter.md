# REFACTOR: Clean up Mastra AI pipeline — eliminate pdf-parse, remove slop, fix bugs

## Problem

The Mastra AI pipeline has multiple issues: dead code, a runtime bug, env var mismatch, unnecessary deps, and — most critically — a **sloppy system design** that extracts PDF text client-side via `pdf-parse` when the LLM can process PDFs directly. This adds an unnecessary dependency, an unnecessary workflow step, and an unnecessary tool file.

---

## Key Design Decision: Eliminate `pdf-parse`

### Current flow (5 steps, 3 files for pipeline logic)

```
startStep → fetchStep (fetch PDF → Buffer) → extractStep (pdf-parse → text) → summarizeStep (text → LLM) → persistStep
```

Files: `summarize-estimate.ts`, `pdf-text-extractor.ts`, `pdf-fetcher.ts`

### Proposed flow (3 steps, 1 file for pipeline logic)

```
startStep → summarizeStep (fetch PDF → send as file to LLM) → persistStep
```

Files: `summarize-estimate.ts` only

### Why this works

The AI SDK supports `type: 'file'` content parts with `mediaType: 'application/pdf'`. The agent already uses `createOpenAI` with Kilo gateway routing to `openai/gpt-5.4-mini` — an OpenAI model that supports PDF file inputs natively. The data can be passed as a URL (which Vercel Blob already provides) or as base64.

**Benefits:**
- **Eliminates `pdf-parse` dependency** — no native build issues, no serverless compat concerns (`pdf-parse` pulls in `@napi-rs/canvas`)
- **Eliminates `pdf-text-extractor` tool** — dead file deleted
- **Eliminates `pdf-fetcher` tool** — already dead, still deleted
- **Eliminates `fetchStep` and `extractStep`** — 2 of 5 workflow steps removed
- **Better quality** — OpenAI's PDF processing handles layout, tables, and scanned text better than `pdf-parse` (which only extracts raw text, losing structure)
- **Simpler error handling** — fewer steps = fewer failure points
- **Smaller bundle** — `pdf-parse` + `pdfjs-dist` + `@napi-rs/canvas` = significant bundle size removed

### Why NOT keep pdf-parse as a fallback?

- `pdf-parse` depends on `@napi-rs/canvas` (native binary) — known to cause issues on Vercel serverless
- It only extracts raw text — loses tables, layout, headers that  inspection reports rely on
- The LLM receiving the PDF directly processes it with vision capabilities, producing better summaries
- Adding complexity "just in case" is the definition of slop — YAGNI

## Execution Plan

### Step 1 — Rewrite workflow to send PDF directly to LLM

Replace `src/mastra/workflows/summarize-estimate.ts` entirely:

```typescript
import { Workflow, createStep } from '@mastra/core/workflows';
import { z } from 'zod';
import { db } from '@/db';
import { estimateRequestTable } from '@/features/estimate/db/schema';
import { eq } from 'drizzle-orm';

const startStep = createStep({
  id: 'start-processing',
  inputSchema: z.object({ estimateRequestId: z.string(), fileUrl: z.string().url() }),
  outputSchema: z.object({ estimateRequestId: z.string(), fileUrl: z.string().url() }),
  execute: async ({ inputData }) => {
    await db.update(estimateRequestTable)
      .set({ status: 'processing' })
      .where(eq(estimateRequestTable.id, inputData.estimateRequestId));
    return { estimateRequestId: inputData.estimateRequestId, fileUrl: inputData.fileUrl };
  }
});

const summarizeStep = createStep({
  id: 'summarize-document',
  inputSchema: z.object({ estimateRequestId: z.string(), fileUrl: z.string().url() }),
  outputSchema: z.object({ estimateRequestId: z.string(), summary: z.string() }),
  execute: async ({ inputData, mastra }) => {
    const agent = mastra.getAgent('estimate-summarizer');

    const result = await agent.generate([
      {
        role: 'user',
        content: [
          { type: 'text', text: 'Provide a structured technical summary of repairs and costs from this report.' },
          { type: 'file', mediaType: 'application/pdf', data: new URL(inputData.fileUrl) },
        ],
      }
    ]);

    return { estimateRequestId: inputData.estimateRequestId, summary: result.text };
  }
});

const persistStep = createStep({
  id: 'persist-result',
  inputSchema: z.object({ estimateRequestId: z.string(), summary: z.string() }),
  outputSchema: z.object({ success: z.boolean() }),
  execute: async ({ inputData }) => {
    await db.update(estimateRequestTable)
      .set({ summary: inputData.summary, status: 'completed' })
      .where(eq(estimateRequestTable.id, inputData.estimateRequestId));
    return { success: true };
  }
});

export const summarizeEstimateWorkflow = new Workflow({
  id: 'summarize-estimate',
  inputSchema: z.object({
    estimateRequestId: z.string(),
    fileUrl: z.string().url(),
  }),
  outputSchema: z.object({ success: z.boolean() }),
})
  .then(startStep)
  .then(summarizeStep)
  .then(persistStep)
  .commit();
```

**Key changes:**
- Removed `fetchStep` — no need to fetch the PDF ourselves; the AI SDK fetches URLs natively
- Removed `extractStep` — no text extraction; LLM processes PDF directly
- `summarizeStep` sends PDF as `{ type: 'file', mediaType: 'application/pdf', data: new URL(inputData.fileUrl) }` — the AI SDK fetches the URL and sends it to the model
- Added `.commit()` — was missing, caused runtime failure
- Schema uses `z.string().url()` throughout instead of `z.instanceof(Buffer)` — no binary data passed between steps
