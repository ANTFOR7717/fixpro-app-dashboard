## Existing Pattern Audit

### Direct SDK usage
The project uses Drizzle ORM directly (`import { db } from "@/db"`), Vercel Blob directly (`import("@vercel/blob")`), and Resend directly. No wrapper/adapter layer around any of these. Mastra should be used the same way — `import { mastra } from "@/mastra"` directly in server actions when needed.

### Next.js config (`next.config.ts`)
Currently an empty config object with no type annotation. Needs `serverExternalPackages` for Mastra.

### Environment variables (`.env.example`)
Grouped by comment headers. AI config goes at the bottom.

### Package manager
Project uses `pnpm` (has `pnpm-lock.yaml`). CLI invocation must use `pnpm dlx`.

### Zod version
Project uses Zod v4 (`^4.2.1`). Mastra's transitive dependency `@ai-sdk/ui-utils` and `@ai-sdk/provider-utils` declare a peer dependency on `zod@^3.23.8`. This causes a peer dependency warning but does not block installation or runtime. The project already has `overrides` in `package.json` for `esbuild` — a Zod override may be needed if runtime issues arise.

## Execution Plan

### Step 1 — Run `mastra init` CLI

**Command:**
```bash
pnpm dlx mastra@latest init --dir src --llm openai --no-example
```

**Flags:**
- `--dir src` — places Mastra files in `src/mastra/`, matching the project's existing `src/` structure
- `--llm openai` — sets OpenAI as the default provider
- `--no-example` — skips example weather agent/tool boilerplate. Also skips subdirectory creation — those are only generated when example code is included.

**What the CLI creates:**
- `src/mastra/index.ts` — Mastra entrypoint with config

**What must be created manually after the CLI:**
- `src/mastra/agents/` — empty directory for agent definitions
- `src/mastra/tools/` — empty directory for tool definitions
- `src/mastra/workflows/` — empty directory for workflow definitions

**What the CLI installs:**
- `@mastra/core`
- `@ai-sdk/openai`
- `ai`

### Step 2 — Review CLI-generated `src/mastra/index.ts`

After CLI runs, inspect the generated `index.ts`. If it includes memory/storage config referencing a local SQLite file (e.g. `mastra.db`), that must be removed or reconfigured — Vercel's serverless filesystem is ephemeral. The project already has Postgres via `pg` which can serve as Mastra's storage backend if needed.

### Step 3 — Update `next.config.ts`

Add `serverExternalPackages` for Mastra.

### Step 4 — Update `.gitignore`

Add `.mastra/` build cache directory.

### Step 5 — Update `.env.example`

Document `OPENAI_API_KEY`. Add actual key to `.env` / `.env.local`.

## File-by-File Changes

### `src/mastra/*`

**Action:** CLI creates `index.ts`. Subdirectories (`agents/`, `tools/`, `workflows/`) created manually via `mkdir -p` + `.gitkeep`.
**Why:** Official scaffolding command for existing projects — installs deps and generates entrypoint. `--no-example` skips subdirectory creation since there's no example code to populate them.
**Impact:** Creates `src/mastra/` with `index.ts` and three empty subdirectories ready for future agent/tool/workflow definitions.

#### Before

Directory does not exist yet.

#### After

- `src/mastra/index.ts` — generated by CLI, reviewed for local storage references (none found)
- `src/mastra/agents/.gitkeep` — created manually
- `src/mastra/tools/.gitkeep` — created manually
- `src/mastra/workflows/.gitkeep` — created manually

#### Reasoning
- The `mastra init` CLI is the documented installation path for existing projects
- `--no-example` prevents the weather agent/tool boilerplate — we don't want dead code
- `--no-example` also means the CLI does not create subdirectories — they must be created manually
- `.gitkeep` files ensure git tracks the empty directories
- Not using `--default` because it force-enables examples; `--dir src --llm openai --no-example` covers all three prompted values (directory, provider, examples) explicitly, ensuring fully non-interactive execution

---

### `next.config.ts`

**Action:** Modify
**Why:** Mastra packages use Node.js APIs and dynamic module loading incompatible with webpack bundling. `serverExternalPackages` tells Next.js to resolve them at runtime.
**Impact:** Build behavior change for `@mastra/*` packages only. No effect on other dependencies.

#### Before

```typescript

const nextConfig  = {
  /* config options here */
};

export default nextConfig;
```

#### After

```typescript
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  serverExternalPackages: ["@mastra/*"],
};

export default nextConfig;
```

#### Reasoning
- `serverExternalPackages` is the standard Next.js mechanism documented in the [Mastra Next.js guide](https://mastra.ai/guides/getting-started/next-js)
- Added `NextConfig` type import for type safety
- Removed placeholder comment

---

### `.gitignore`

**Action:** Modify
**Why:** Mastra generates a `.mastra/` build cache directory when running `mastra dev` or `mastra build`. This is build output and should not be committed.
**Impact:** Prevents build artifacts from entering version control.

#### Before

```text
# typescript
*.tsbuildinfo
next-env.d.ts
```

#### After

```text
# typescript
*.tsbuildinfo
next-env.d.ts

# mastra
.mastra/
```

#### Reasoning
- `.mastra/` is equivalent to `.next/` — build output, not source. The project already ignores `/.next/`.
- Appended at the end following the existing pattern of grouped entries with comment headers.

---

### `.env.example`

**Action:** Modify
**Why:** Documents `OPENAI_API_KEY` which `@ai-sdk/openai` reads from `process.env` automatically
**Impact:** Documentation only. No runtime effect.

#### Before

```text
# Vercel Blob
BLOB_READ_WRITE_TOKEN=your-vercel-blob-read-write-token
```

#### After

```text
# Vercel Blob
BLOB_READ_WRITE_TOKEN=your-vercel-blob-read-write-token

# AI Provider
OPENAI_API_KEY=your-openai-api-key
```

#### Reasoning
- `@ai-sdk/openai` reads `OPENAI_API_KEY` automatically — no code config needed
- Follows existing grouping convention
- The `mastra init` CLI has a `--llm-api-key` flag that writes directly to `.env`, but we skip it to avoid overwriting the existing `.env` file

---

### `package.json` (CLI-modified)

**Action:** Modify (by CLI)
**Why:** `mastra init` runs dependency installation, adding `@mastra/core`, `@ai-sdk/openai`, and `ai` to `dependencies`
**Impact:** Three new runtime dependencies added. Zod v3 peer dependency warning will appear — does not block installation.

#### Before

Not shown — CLI handles this automatically.

#### After

Not shown — CLI handles this automatically. Expected additions to `dependencies`:
```json
"@mastra/core": "^1.x",
"@ai-sdk/openai": "^x.x",
"ai": "^x.x"
```

#### Reasoning
- The CLI manages version resolution — pinning versions manually would drift from what the CLI installs
- The Zod v3 vs v4 peer dependency mismatch (`✕ unmet peer zod@^3.23.8: found 4.4.3`) is a known compatibility gap between `ai` SDK and Zod v4. It produces warnings but does not cause runtime failures for basic agent usage. If runtime issues arise, a `pnpm.overrides` entry for Zod can resolve it.
